Summary — Tất cả câu hỏi ôn tập

138 câu · 7 chủ đề · Mỗi câu 1 dòng hỏi + 1 dòng đáp

01 · Python Core — 20 câu
01GIL là gì và tại sao nó tồn tại?Khóa toàn cục chỉ cho một luồng chạy tại một thời điểm, để bảo vệ bộ đếm tham chiếu quản lý bộ nhớ khỏi race condition.
02GIL ảnh hưởng thế nào đến đa luồng?Tác vụ I/O nhả GIL nên đa luồng có hiệu quả; tác vụ CPU không song song được — dùng đa tiến trình thay.
03Phân biệt đa luồng, đa tiến trình và asyncio?Đa luồng: chia sẻ bộ nhớ, bị GIL. Đa tiến trình: GIL riêng, tốt cho CPU. Asyncio: một luồng, chuyển ngữ cảnh khi chờ I/O.
04Async và await hoạt động như thế nào?async tạo coroutine; await là điểm tạm dừng nhường quyền cho event loop xử lý coroutine khác.
05asyncio.gather vs asyncio.wait khác gì nhau?gather chạy tất cả, trả kết quả theo thứ tự, một lỗi là fail toàn bộ; wait linh hoạt hơn, chia kết quả thành done/pending.
06Generator là gì? Khi nào nên dùng?Hàm dùng yield tạo giá trị theo yêu cầu; dùng khi cần tiết kiệm bộ nhớ với dữ liệu lớn hoặc stream.
07__slots__ dùng để làm gì?Khai báo cố định thuộc tính của class, bỏ __dict__ nội bộ, tiết kiệm 40–60% bộ nhớ khi tạo hàng triệu object.
08Decorator là gì? Cách hoạt động?Hàm nhận hàm khác làm đầu vào, bọc thêm logic, trả về hàm mới — dùng cú pháp @tên để áp dụng.
09functools.wraps dùng để làm gì?Sao chép tên, docstring của hàm gốc sang hàm bọc trong decorator để log và debug không bị nhầm tên.
10Context manager là gì? Tại sao cần thiết?Cú pháp "with" đảm bảo tài nguyên luôn được dọn dẹp đúng cách dù có lỗi hay không.
11Phân biệt deepcopy và shallow copy?Shallow copy chỉ sao chép tầng ngoài, object con vẫn dùng chung; deepcopy tạo bản sao hoàn toàn độc lập đệ quy.
12List comprehension vs hàm map() — dùng khi nào?List comprehension rõ ràng, kết quả ngay; map() lười biếng, tiết kiệm bộ nhớ khi duyệt qua một lần với chuỗi dài.
13*args và **kwargs là gì?*args gom đối số vị trí thừa vào tuple; **kwargs gom đối số có tên thừa vào dict — dùng nhiều trong decorator.
14Python quản lý bộ nhớ như thế nào?Ba cơ chế: đếm tham chiếu (chính), garbage collector cho vòng tham chiếu, và bể bộ nhớ tái sử dụng object nhỏ.
15is và == khác nhau thế nào?== so sánh giá trị; is so sánh danh tính (cùng object trong bộ nhớ). Chỉ dùng is để kiểm tra None/True/False.
16Exception chaining (raise … from …) là gì?Gắn lỗi gốc vào lỗi mới, giữ nguyên nguyên nhân kỹ thuật trong khi cung cấp lỗi có ý nghĩa nghiệp vụ hơn.
17Phân biệt classmethod, staticmethod và instance method?instance method nhận self; classmethod nhận cls (factory); staticmethod không nhận gì — hàm tiện ích đặt trong class.
18__init__ và __new__ khác nhau thế nào?__new__ cấp phát bộ nhớ tạo object rỗng; __init__ điền dữ liệu vào. Thường chỉ override __init__, __new__ dùng cho Singleton.
19Pydantic dùng để làm gì? Tại sao quan trọng trong API tài chính?Tự động validate và chuyển đổi dữ liệu qua type hints; ngăn dữ liệu sai lan vào hệ thống và tự sinh tài liệu API.
20Làm sao tối ưu performance cho Python service nhiều request?Asyncio, connection pooling, cache Redis, generator cho dữ liệu lớn, đo trước khi tối ưu, tác vụ nặng dùng Celery.
02 · FastAPI — 20 câu
01FastAPI lifespan hoạt động thế nào? Tại sao không dùng on_event?Hàm async generator chia đôi bằng yield: trước yield là khởi động, sau yield là dọn dẹp; on_event đã deprecated vì thiếu đảm bảo dọn dẹp khi lỗi.
02Dependency Injection trong FastAPI là gì?FastAPI tự động gọi hàm phụ thuộc và truyền kết quả vào endpoint; cùng request chỉ gọi một lần dù nhiều endpoint cần.
03Middleware là gì? Thứ tự thực thi thế nào?Code chạy với mọi request/response; middleware thêm sau sẽ chạy trước — như lớp hành tây, ngoài cùng vào trước.
04Phân biệt HTTPException, custom exception handler và RequestValidationError?HTTPException: ném thủ công; custom handler: bắt lỗi tự định nghĩa, trả JSON nhất quán; RequestValidationError: Pydantic tự ném khi dữ liệu sai.
05Background Tasks vs Celery — khi nào dùng cái nào?Background Tasks: đơn giản, mất khi tắt server; Celery: bền vững, tự retry, có monitoring — dùng cho tác vụ quan trọng.
06Hàm async và sync trong FastAPI — khi nào dùng cái nào?async khi gọi hàm bất đồng bộ; sync khi dùng thư viện blocking — FastAPI tự chạy sync trong thread riêng để không chặn event loop.
07FastAPI dùng ASGI hay WSGI? Khác nhau thế nào?FastAPI dùng ASGI (qua Starlette) — hỗ trợ async và WebSocket; WSGI là giao thức cũ đồng bộ của Flask/Django. Server chạy là Uvicorn.
08APIRouter dùng để làm gì?Chia API thành module theo domain, mỗi router có prefix, tags và dependency riêng — app chỉ include router, không quản lý từng endpoint.
09Response Model dùng để làm gì?Khai báo cấu trúc response, FastAPI tự lọc chỉ giữ field khai báo — bảo vệ dữ liệu nhạy cảm và tạo tài liệu chính xác.
10Làm sao implement JWT authentication trong FastAPI?Khai báo OAuth2PasswordBearer lấy token từ header → hàm dependency giải mã xác minh → endpoint khai báo Depends vào hàm đó.
11Làm sao handle CORS trong FastAPI?Thêm CORSMiddleware, liệt kê domain cụ thể, bật allow_credentials nếu cần — không dùng wildcard * cho API có xác thực.
12Path param, Query param và Request body — khi nào dùng?Path: định danh resource bắt buộc; Query: lọc/phân trang tùy chọn; Body: dữ liệu phức tạp/nhạy cảm với POST/PUT/PATCH.
13Làm sao test FastAPI endpoint?Dùng TestClient không cần server thật; override Depends để inject mock/in-memory DB; test cả trường hợp thành công và lỗi.
14Streaming response là gì? Khi nào dùng?Server gửi dữ liệu từng phần khi có, client nhận ngay — dùng cho file lớn, CSV xuất, hoặc phản hồi LLM theo từng token.
15WebSocket trong FastAPI hoạt động thế nào?Kết nối hai chiều liên tục, server có thể chủ động push — dùng cho thông báo realtime, tỷ giá, dashboard giao dịch trực tiếp.
16Làm sao implement rate limiting trong FastAPI?Dùng Redis làm bộ đếm nguyên tử theo IP+thời gian; từ chối 429 khi vượt ngưỡng — Redis đảm bảo chính xác khi nhiều server.
17Làm sao trả về nhiều loại response từ một endpoint?Khai báo responses={200: ..., 400: ...} trong decorator; trả về JSONResponse với status_code phù hợp để bypass response model mặc định.
18OpenAPI docs được sinh tự động như thế nào trong FastAPI?FastAPI đọc type hints, Pydantic models, metadata decorator → tạo JSON OpenAPI → Swagger UI (/docs) và ReDoc (/redoc) tự động.
19Làm sao xử lý file upload trong FastAPI?Nhận multipart/form-data, kiểm tra loại và kích thước, stream trực tiếp lên S3 — không đọc toàn bộ vào RAM.
20Làm sao implement health check endpoint đúng cách?Liveness: phản hồi nhanh xác nhận còn sống; Readiness: kiểm tra DB/Redis/dependencies trước khi nhận traffic mới từ Kubernetes.
03 · Database — 20 câu
01ACID là gì? Giải thích từng tính chất?Atomicity: all or nothing; Consistency: luôn hợp lệ; Isolation: giao dịch không thấy nhau; Durability: commit là vĩnh viễn.
02Clustered Index và Non-Clustered Index khác nhau thế nào?Clustered: dữ liệu sắp xếp vật lý theo index, một bảng chỉ có một; Non-Clustered: index riêng lưu con trỏ đến dữ liệu, tìm hai bước.
03Khi nào nên tạo index? Khi nào không nên?Nên: cột WHERE/JOIN/ORDER BY có nhiều giá trị khác nhau; Không nên: bảng nhỏ, cột ít giá trị (giới tính), bảng ghi nhiều đọc ít.
04Composite Index là gì? Rule thiết kế ra sao?Index nhiều cột; đặt cột equality trước, cột range sau; index (A,B) hỗ trợ query dùng A nhưng không hỗ trợ chỉ dùng B.
05Transaction Isolation Level — có những mức nào?Read Uncommitted → Read Committed (mặc định PG) → Repeatable Read → Serializable. Ngân hàng thường dùng Repeatable Read hoặc Serializable.
06Dirty Read, Non-repeatable Read, Phantom Read là gì?Dirty: đọc dữ liệu chưa commit; Non-repeatable: cùng hàng đọc hai lần cho kết quả khác; Phantom: cùng điều kiện lần hai thấy thêm hàng mới.
07Optimistic Lock và Pessimistic Lock — khi nào dùng?Pessimistic: khóa ngay khi đọc, chắc chắn nhưng dễ deadlock — dùng cho số dư; Optimistic: kiểm tra version khi ghi — dùng khi ít xung đột.
08N+1 Query Problem là gì? Cách khắc phục?Lấy N bản ghi rồi loop gọi thêm N query riêng lẻ; khắc phục: dùng một query WHERE IN để lấy hết dữ liệu liên quan, ghép trong code.
09EXPLAIN ANALYZE dùng để làm gì?Hiển thị kế hoạch thực thi thật của PostgreSQL: có dùng index không, quét bao nhiêu hàng, thời gian từng bước — để tìm query chậm.
10Redis dùng khi nào trong hệ thống backend?Cache kết quả tốn kém, lưu session, rate limiting bộ đếm, distributed lock đảm bảo chỉ một server xử lý một giao dịch cùng lúc.
11Cache-Aside và Write-Through — khác nhau thế nào?Cache-Aside: chỉ vào cache khi được hỏi lần đầu (lazy); Write-Through: ghi cả DB lẫn cache đồng thời — luôn fresh nhưng ghi chậm hơn.
12Cache Stampede là gì? Cách xử lý?Nhiều request cùng thấy cache miss, đồng loạt query DB; giải pháp: distributed lock cho phép chỉ một request rebuild cache, còn lại chờ.
13Tại sao không dùng số thực để lưu tiền tệ?Float không biểu diễn chính xác số thập phân trong nhị phân (0.1+0.2≠0.3); dùng NUMERIC trong DB và Decimal trong Python.
14Sharding và Partitioning khác nhau thế nào?Partitioning: chia bảng trong cùng một server, ứng dụng không cần biết; Sharding: chia ra nhiều server khác nhau, ứng dụng phải biết đến server nào.
15Deadlock là gì? Làm sao tránh?Hai giao dịch chờ nhau giải phóng khóa vô hạn; phòng tránh bằng cách luôn khóa tài nguyên theo cùng một thứ tự cố định.
16Connection Pool là gì? Tại sao quan trọng?Tạo sẵn tập kết nối DB, request mượn dùng rồi trả — tránh tốn 100–500ms tạo kết nối mới mỗi lần và không vượt giới hạn max connections.
17UUID và Auto-increment ID — khi nào dùng cái nào?Auto-increment: nhỏ, nhanh, dễ đoán; UUID v4: an toàn, toàn cầu duy nhất; UUID v7: kết hợp cả hai — tăng theo thời gian và toàn cầu duy nhất.
18Immutable Ledger Pattern là gì? Tại sao dùng trong fintech?Không bao giờ sửa xóa giao dịch, chỉ thêm bản ghi mới; đảm bảo lịch sử đầy đủ bất biến, dễ kiểm toán, không mất bằng chứng.
19Tại sao cần Idempotency Key trong payment API?Khi mạng lỗi client retry, key duy nhất đảm bảo server nhận ra request trùng lặp và trả kết quả cũ — giao dịch chỉ thực hiện một lần.
20CAP Theorem là gì? Áp dụng thực tế thế nào?Hệ thống phân tán chỉ đảm bảo được 2/3: Consistency, Availability, Partition Tolerance; ngân hàng ưu tiên C (nhất quán) hơn A (sẵn sàng).
04 · Bảo mật — 20 câu
01Phân biệt Authentication và Authorization?Authentication: xác nhận bạn là ai; Authorization: kiểm tra bạn được làm gì — phải xác thực xong mới phân quyền được.
02Cấu trúc của JWT là gì? Làm sao hoạt động?Header.Payload.Signature — server xác minh bằng cách tính lại Signature từ Header+Payload; payload sai thì chữ ký không khớp.
03Access Token và Refresh Token khác nhau thế nào?Access Token sống ngắn (15–60 phút), gửi mọi request; Refresh Token sống dài (7–30 ngày), chỉ dùng để xin Access Token mới, lưu httpOnly cookie.
04Làm sao logout khi dùng JWT? Vấn đề là gì?JWT không thu hồi được; giải pháp: lưu jti vào Redis blacklist khi logout, mỗi request kiểm tra jti có trong blacklist không.
05OAuth 2.0 Authorization Code Flow là gì?Redirect đến provider → user đồng ý → nhận code ngắn hạn → server đổi code lấy token server-to-server (token không lộ qua browser).
06SQL Injection là gì? Cách phòng tránh?Kẻ tấn công chèn SQL qua input; phòng bằng parameterized query — dữ liệu luôn được xử lý như dữ liệu, không bao giờ như SQL.
07XSS là gì? Cách phòng tránh?Chèn JavaScript độc hại vào trang; phòng bằng escape HTML khi hiển thị, Content-Security-Policy, lưu token trong httpOnly cookie.
08CSRF là gì? Cách phòng tránh?Lừa browser tự gửi request kèm cookie hợp lệ; phòng bằng SameSite=Strict/Lax trên cookie, hoặc CSRF token nhúng vào form.
09Tại sao không dùng MD5/SHA-1 cho mật khẩu? Dùng gì thay?MD5/SHA-1 quá nhanh, dễ brute force; dùng bcrypt, Argon2 hoặc scrypt — cố tình chậm, tốn bộ nhớ, tự thêm salt.
10HTTPS và TLS hoạt động thế nào?TLS bắt tay: server gửi chứng chỉ, hai bên thỏa thuận khóa bí mật chung không qua mạng → mọi dữ liệu được mã hóa bằng khóa đó.
11OWASP Top 10 là gì? Kể vài lỗ hổng quan trọng?Top 2021: #1 Broken Access Control, #2 Cryptographic Failures, #3 Injection — với fintech, Broken Access Control nguy hiểm nhất.
12Rate limiting — các chiến lược nào?Fixed Window: đơn giản nhưng có burst cuối window; Sliding Window: chính xác hơn; Token Bucket: cho phép burst tự nhiên — phổ biến nhất.
13Secrets Management là gì? Cách lưu trữ thông tin nhạy cảm?Không commit vào git; dùng biến môi trường hoặc file .env; production dùng AWS Secrets Manager/HashiCorp Vault — có audit log và xoay vòng tự động.
14API Key và JWT khác nhau như thế nào?API Key: chuỗi đơn giản, server tra DB mỗi request, thu hồi ngay được — dùng cho machine-to-machine; JWT: tự chứa thông tin, nhanh hơn — dùng cho user.
15Principle of Least Privilege là gì?Mỗi thực thể chỉ được đúng quyền tối thiểu cần thiết; khi bị xâm phạm, thiệt hại bị giới hạn trong phạm vi quyền của component đó.
16Security Headers là gì? Tại sao quan trọng?Header HTTP hướng dẫn browser tăng bảo vệ: CSP ngăn XSS, X-Frame-Options ngăn clickjacking, HSTS ép HTTPS — lớp phòng thủ bổ sung.
17Audit logging là gì? Tầm quan trọng trong fintech?Ghi ai làm gì khi nào từ đâu kết quả gì; bắt buộc theo pháp lý, dùng để điều tra tranh chấp và kiểm toán — không ai được xóa.
18Defense in Depth là gì?Xây nhiều lớp bảo vệ độc lập: HTTPS → JWT → phân quyền → validate input → parameterized query → giới hạn nghiệp vụ → audit log.
19mTLS là gì? Khi nào dùng?TLS hai chiều — cả client cũng xác thực chứng chỉ; dùng cho giao tiếp giữa microservice, kết nối DB cần bảo mật tối đa.
20Làm sao validate và sanitize input đúng cách?Không tin dữ liệu từ bên ngoài; Pydantic tự validate kiểu/range; thêm validate nghiệp vụ (số tiền không âm); sanitize HTML để chống XSS.
05 · Thuật Toán & CTDL — 20 câu
01Big O Notation là gì? Tại sao quan trọng?Diễn đạt tốc độ tăng trưởng thời gian/bộ nhớ khi dữ liệu lớn lên; giúp chọn đúng thuật toán trước khi code ra quy mô lớn.
02Array và Linked List khác nhau thế nào?Array: truy cập O(1), chèn xóa O(n); Linked List: chèn xóa O(1), tìm theo vị trí O(n). Python list là array động đủ dùng hầu hết.
03Stack và Queue là gì? Ứng dụng thực tế?Stack LIFO: lịch sử browser, call stack Python; Queue FIFO: hàng đợi in, Celery task queue, xử lý sự kiện theo thứ tự.
04Hash Table hoạt động thế nào? Collision xử lý ra sao?Hàm băm ánh xạ key → vị trí mảng, O(1) trung bình; collision xử lý bằng Chaining (danh sách tại ô) hoặc Open Addressing (tìm ô trống kế).
05Tìm kiếm nhị phân hoạt động thế nào?Mảng đã sắp xếp, mỗi bước so với phần tử giữa rồi bỏ nửa không chứa đáp án — O(log n), 1 triệu phần tử tìm trong 20 bước.
06BFS và DFS khác nhau thế nào? Khi nào dùng?BFS dùng Queue, duyệt theo tầng — tìm đường ngắn nhất; DFS dùng Stack/đệ quy, đi sâu — duyệt toàn bộ, phát hiện chu trình.
07Merge Sort và Quick Sort khác nhau thế nào?Merge Sort: O(n log n) mọi trường hợp, cần O(n) thêm; Quick Sort: O(n log n) trung bình, O(n²) xấu nhất, sắp xếp tại chỗ, nhanh hơn thực tế.
08Heap và Priority Queue là gì?Heap là cây nhị phân đặc biệt: phần tử nhỏ/lớn nhất luôn ở gốc O(1); Priority Queue là cấu trúc trừu tượng, Heap là cách implement tối ưu.
09LRU Cache là gì? Implement thế nào?Loại phần tử ít dùng gần đây nhất khi đầy; implement bằng OrderedDict Python — get/put đều O(1) nhờ xóa rồi chèn lại cuối.
10Dynamic Programming là gì? Nguyên tắc hoạt động?Chia bài toán lớn thành bài toán con chồng nhau, giải một lần, lưu kết quả, tái sử dụng — Fibonacci từ O(2ⁿ) xuống O(n).
11Giải bài Two Sum trong O(n)?Duyệt mảng, mỗi phần tử tính phần bù (target - x), tra hash table xem đã gặp chưa, lưu vào nếu chưa — O(n) thời gian, O(n) bộ nhớ.
12Giải bài Valid Parentheses dùng Stack thế nào?Push ngoặc mở; gặp ngoặc đóng thì pop và kiểm tra khớp; cuối cùng stack rỗng là hợp lệ — O(n) thời gian, O(n) bộ nhớ.
13Sliding Window là kỹ thuật gì?Duy trì cửa sổ kích thước k trượt qua mảng: thêm phần tử mới, bỏ phần tử cũ O(1) mỗi bước — tổng O(n) thay vì O(n×k).
14Làm sao đảo ngược Linked List?Ba con trỏ prev/curr/next: lưu next, đổi hướng curr→prev, tiến cả hai — O(n) thời gian, O(1) bộ nhớ.
15Floyd's Cycle Detection là gì?Con rùa bước 1, con thỏ bước 2; có chu trình thì thỏ gặp rùa; không có thì thỏ ra null — O(1) bộ nhớ thay vì O(n) của hash set.
16Cây nhị phân tìm kiếm (BST) là gì? Duyệt thế nào?Trái < cha < phải; Inorder cho dãy tăng, Preorder để sao chép, Postorder để xóa; mất cân bằng thành O(n) — dùng AVL/Red-Black.
17Topological Sort là gì? Ứng dụng?Sắp xếp DAG sao cho mọi cạnh u→v thì u trước v; dùng cho thứ tự build module, task dependency, phát hiện vòng phụ thuộc.
18Trie là gì? Khi nào dùng?Cây tiền tố lưu chuỗi; tìm/kiểm tra prefix O(độ dài chuỗi) bất kể có bao nhiêu chuỗi — dùng cho autocomplete, từ điển, keyword filter.
19Graph — Adjacency Matrix vs Adjacency List?Matrix: kiểm tra cạnh O(1), tốn O(n²) bộ nhớ — dùng cho đồ thị dày; List: O(n+E) bộ nhớ — phổ biến hơn với đồ thị thưa.
20Khi nào không dùng đệ quy? Chuyển sang vòng lặp thế nào?Khi đệ quy sâu quá 1000 cấp (Python limit); chuyển bằng cách dùng stack tự quản lý trong heap — bộ nhớ heap lớn hơn call stack nhiều.
06 · Project & System Design — 20 câu
01Hãy giới thiệu project gần đây nhất của bạn?Dùng cấu trúc STAR-light: tình huống → nhiệm vụ → hành động cụ thể → kết quả có con số; tránh nói chung chung không có context.
02Monolith và Microservices — khi nào chọn cái nào?Monolith: team nhỏ, giai đoạn đầu, cần tốc độ; Microservices: team lớn, các module scale khác nhau, domain đã rõ ràng — đừng chọn sớm.
03Circuit Breaker Pattern là gì? Tại sao cần?Ngăn cascading failure khi service phụ thuộc hỏng; ba trạng thái: CLOSED → OPEN (ngắt mạch, fail nhanh) → HALF-OPEN (thử lại dần).
04Saga Pattern là gì? Khi nào dùng trong fintech?Chuỗi local transaction có compensating transaction khi lỗi; Choreography (phân tán) vs Orchestration (điều phối trung tâm) — fintech dùng Orchestration.
05High Availability là gì? Làm sao đạt được?Hệ thống tiếp tục hoạt động khi thành phần hỏng; đạt bằng: loại điểm thất bại đơn, chạy nhiều instance, load balancer, failover tự động.
06Horizontal scaling và Vertical scaling khác nhau thế nào?Vertical: nâng cấp máy, có giới hạn, đơn giản; Horizontal: thêm máy, gần như vô hạn, cần stateless — thực tế kết hợp cả hai.
07Load Balancing hoạt động thế nào? Các thuật toán phân phối?Phân phối request đến nhiều instance; Round Robin (tuần tự), Least Connections (ít nhất đang xử lý), Sticky Sessions (cùng user cùng server).
08Blue/Green deployment và Canary deployment khác nhau thế nào?Blue/Green: hai môi trường song song, chuyển toàn bộ traffic — rollback cực nhanh; Canary: tăng dần %, theo dõi metrics — tiết kiệm tài nguyên hơn.
09Observability là gì? Metrics, Logs, Traces khác nhau thế nào?Metrics: số đo theo thời gian, cảnh báo; Logs: sự kiện rời rạc, điều tra; Traces: hành trình request qua nhiều service, tìm bottleneck.
10Làm sao thiết kế API versioning?URL path phổ biến nhất (/v1, /v2); thêm field không phá tương thích, xóa/đổi kiểu mới cần version mới; giữ version cũ đủ lâu để migrate.
11Message Queue là gì? Khi nào dùng thay vì gọi API trực tiếp?Buffer giữa producer và consumer; dùng khi tốc độ khác nhau, cần độ tin cậy khi consumer tạm hỏng, hoặc muốn decouple hai service.
12CAP Theorem ảnh hưởng đến thiết kế hệ thống ngân hàng thế nào?Partition Tolerance bắt buộc → phải chọn C hoặc A; ngân hàng chọn Consistency (CP) — thà từ chối giao dịch còn hơn số dư sai.
13Làm sao thiết kế hệ thống thanh toán chịu tải cao?Idempotency Key chống trùng lặp, Immutable Ledger không sửa xóa, queue tiếp nhận nhanh → worker xử lý có kiểm soát, cache tỷ giá/giới hạn.
14Event-driven Architecture là gì?Service giao tiếp qua sự kiện thay vì gọi trực tiếp; dễ thêm tính năng mới (thêm consumer), nhưng khó debug và cần message broker.
15Làm sao xử lý nợ kỹ thuật trong dự án?Đo lường và ưu tiên; áp dụng Boy Scout Rule mỗi lần đụng code; hệ thống lớn dùng Strangler Fig Pattern thay thế dần không rewrite toàn bộ.
16SLI, SLO, SLA là gì?SLI: số đo thực tế (tỷ lệ thành công); SLO: mục tiêu nội bộ team; SLA: cam kết pháp lý với khách hàng — SLA thường dễ hơn SLO để có buffer.
17Làm sao đảm bảo eventual consistency trong hệ thống ngân hàng?Số dư cần strong consistency; dashboard/thông báo chấp nhận chậm vài giây; dùng Outbox Pattern, Reconciliation, Idempotent Consumer.
18Làm sao debug khi hệ thống chậm trong production?Xác định phạm vi → kiểm tra metrics → xem distributed traces → EXPLAIN ANALYZE query chậm → logs; thay đổi một thứ, đo rồi mới tiếp.
19Tại sao chọn FastAPI cho dự án fintech?Async I/O hiệu suất cao, Pydantic validate tự động, tài liệu tự sinh, DI dễ test — nhược điểm: hệ sinh thái nhỏ hơn Django, không có ORM tích hợp.
20Câu hỏi nào bạn muốn hỏi người kỹ thuật ở Tyme?Hỏi về thách thức kỹ thuật lớn nhất, kiến trúc microservice, quy trình incident response, cách đo thành công của engineer sau 6 tháng.
07 · OOP & SOLID — 18 câu
01OOP là gì? Tại sao dùng OOP?Tổ chức code xoay quanh đối tượng gộp dữ liệu và hành vi; dễ hiểu, tái sử dụng, thay đổi cục bộ, mở rộng không phá vỡ cái cũ.
02Encapsulation — Đóng gói là gì?Che giấu dữ liệu bên trong, chỉ truy cập qua phương thức kiểm soát — bảo vệ tính toàn vẹn, biết chính xác ai thay đổi dữ liệu.
03Inheritance — Kế thừa là gì? Vấn đề của kế thừa sâu?Class con tái sử dụng và mở rộng class cha; kế thừa sâu gây thay đổi tầng trên ảnh hưởng toàn bộ — giữ nông, ưu tiên composition.
04Polymorphism là gì? Override và Overload khác nhau?Cùng interface hành xử khác nhau theo đối tượng; Override: ghi đè ở class con (runtime); Overload: cùng tên khác tham số (compile-time, Python dùng default args).
05Abstraction — Trừu tượng là gì?Ẩn sự phức tạp, chỉ lộ interface cần thiết; abstract class/interface định nghĩa "làm gì" không định nghĩa "làm thế nào".
06Abstract Class và Interface khác nhau thế nào?Abstract Class: có code dùng chung, kế thừa một; Interface: hợp đồng thuần túy, implement nhiều — không chắc thì dùng interface, linh hoạt hơn.
07Composition over Inheritance — tại sao ưu tiên composition?"Has-a" thay vì "Is-a"; linh hoạt đổi component lúc runtime, tránh kế thừa sâu, dễ test mock từng phần — kế thừa vẫn hợp lý khi quan hệ thực sự is-a.
08SOLID là gì? Tại sao quan trọng?Năm nguyên tắc OOP của Uncle Bob: SRP, OCP, LSP, ISP, DIP — giúp code dễ thay đổi, dễ test, dễ mở rộng; hiểu lý do hơn áp dụng máy móc.
09S — Single Responsibility Principle là gì?Một class chỉ một lý do để thay đổi; vi phạm khi class vừa xử lý logic vừa gửi email vừa ghi log — tách thành class riêng cho từng trách nhiệm.
10O — Open/Closed Principle là gì?Mở để mở rộng, đóng để sửa đổi; thêm tính năng bằng code mới không sửa code cũ — dùng abstract class + class con mới thay vì thêm if-else.
11L — Liskov Substitution Principle là gì?Class con phải thay thế được class cha mà không phá chương trình; vi phạm khi override để throw "not implemented" hoặc kết quả sai expectation.
12I — Interface Segregation Principle là gì?Không ép class implement phương thức không cần; tách interface lớn thành nhiều interface nhỏ chuyên biệt — class chỉ implement những gì có nghĩa.
13D — Dependency Inversion Principle là gì?Module cấp cao phụ thuộc vào abstraction, không vào implementation cụ thể; đổi database chỉ sửa Repository, không sửa Service.
14Dependency Injection là gì? Các kiểu DI?Truyền dependency từ bên ngoài vào thay vì tự tạo; Constructor Injection (phổ biến nhất), Method Injection, Property Injection.
15Design Pattern là gì? Các nhóm pattern phổ biến?Giải pháp tái sử dụng cho bài toán thiết kế đã kiểm chứng; Creational (Singleton, Factory, Builder), Structural (Adapter, Decorator), Behavioral (Observer, Strategy, Repository).
16Singleton Pattern — khi nào dùng và cạm bẫy?Dùng cho connection pool, config, logger; cạm bẫy: global state khó test, race condition đa luồng — nhiều người ưu tiên DI thay thế.
17Repository Pattern là gì? Tại sao phổ biến trong backend?Tầng trung gian giữa business logic và DB; Service gọi get_by_id thay vì viết SQL; đổi DB chỉ sửa Repository, dễ test bằng InMemoryRepository.
18Làm thế nào nhận biết code đang vi phạm OOP và SOLID?SRP: class tên có "And", file nghìn dòng; OCP: if-else dài thêm loại mới; LSP: class con override để throw "not implemented"; DIP: từ khóa new trong business logic.
