01. GIL là gì và tại sao nó tồn tại?

GIL (Global Interpreter Lock) là một cái khóa trong CPython — tại bất kỳ thời điểm nào, chỉ có một thread được phép chạy Python bytecode, dù máy có nhiều nhân CPU.

GIL tồn tại vì Python dùng reference counting để quản lý bộ nhớ. Mỗi object có bộ đếm số nơi đang trỏ đến nó. Nếu nhiều thread cùng tăng/giảm bộ đếm này đồng thời sẽ gây race condition — object bị xóa nhầm khi đang dùng, hoặc không bao giờ được xóa.

GIL như cái mic duy nhất trong hội trường — chỉ 1 người nói tại 1 thời điểm, tránh tất cả nói cùng lúc gây loạn.
import threading, time

count = 0
def increment():
    global count
    for _ in range(1_000_000):
        count += 1  # GIL bảo vệ thao tác này

threads = [threading.Thread(target=increment) for _ in range(2)]
for t in threads: t.start()
for t in threads: t.join()
# count ~ 2_000_000 (không bị corrupt dù 2 thread)
02. GIL ảnh hưởng thế nào đến lập trình đa luồng?

I/O-bound (gọi API, đọc file, query DB): thread tự nhả GIL khi chờ, thread khác chạy vào. Threading vẫn hiệu quả.

CPU-bound (xử lý ảnh, thuật toán nặng): các thread tranh nhau giữ GIL, chạy tuần tự chứ không song song thật. Dùng threading ở đây không giúp ích, thậm chí chậm hơn. Giải pháp: multiprocessing — mỗi process có GIL riêng, chạy thật sự song song.

import time, threading, multiprocessing

def cpu_task():
    sum(i**2 for i in range(1_000_000))

# Threading — không nhanh hơn vì GIL
start = time.time()
threads = [threading.Thread(target=cpu_task) for _ in range(4)]
for t in threads: t.start()
for t in threads: t.join()
print(f"Threading: {time.time()-start:.2f}s")  # ~4s

# Multiprocessing — thật sự song song
start = time.time()
with multiprocessing.Pool(4) as pool:
    pool.map(lambda _: cpu_task(), range(4))
print(f"Multiprocessing: {time.time()-start:.2f}s")  # ~1s
03. Phân biệt đa luồng, đa tiến trình và asyncio?

Ba cơ chế xử lý đồng thời, mỗi cái phù hợp một use case khác nhau:

# Threading — share memory, bị GIL, tốt cho I/O
from concurrent.futures import ThreadPoolExecutor
with ThreadPoolExecutor(max_workers=10) as ex:
    results = list(ex.map(fetch_url, urls))  # gọi 10 API song song

# Multiprocessing — process riêng, GIL riêng, tốt cho CPU
from concurrent.futures import ProcessPoolExecutor
with ProcessPoolExecutor(max_workers=4) as ex:
    results = list(ex.map(resize_image, images))  # xử lý ảnh song song

# asyncio — 1 thread, event loop, tốt nhất cho I/O concurrency cao
import asyncio
async def main():
    results = await asyncio.gather(*[fetch(url) for url in urls])
    # 1000 request đồng thời trong 1 thread

Quy tắc chọn: I/O ít kết nối → threading; I/O nhiều kết nối → asyncio; tính toán nặng → multiprocessing.

04. Async và await hoạt động như thế nào?

Hàm có async def tạo ra coroutine — không chạy ngay khi gọi, phải được event loop điều phối. Từ khóa await là điểm nhường quyền: hàm tạm dừng, event loop chạy coroutine khác, khi I/O xong thì quay lại tiếp tục.

Như nhân viên phục vụ: ghi order xong không đứng chờ bếp nấu, quay sang phục vụ bàn khác. Khi bếp kêu thì mới quay lại lấy món.
import asyncio

async def fetch(name, delay):
    print(f"  start {name}")
    await asyncio.sleep(delay)   # nhường quyền — không block
    print(f"  done  {name}")
    return f"result-{name}"

async def main():
    # Chạy song song — tổng ~1s thay vì 3s
    results = await asyncio.gather(
        fetch("A", 1),
        fetch("B", 1),
        fetch("C", 1),
    )
    print(results)

asyncio.run(main())

Lưu ý: nếu gọi hàm blocking (time.sleep, requests.get) bên trong async def sẽ block toàn bộ event loop. Phải dùng thư viện async (httpx, asyncpg, aiofiles).

05. Sự khác biệt giữa asyncio.gather và asyncio.wait?

gather: chạy tất cả, chờ tất cả xong, trả kết quả theo đúng thứ tự ban đầu. Nếu 1 cái lỗi mặc định toàn bộ fail. Dùng khi cần đủ kết quả mới làm tiếp.

wait: linh hoạt hơn — có thể dừng khi cái đầu tiên xong, hoặc khi cái đầu tiên lỗi. Trả về 2 set: done và pending, xử lý từng cái riêng.

import asyncio

# gather — chờ tất cả, kết quả theo thứ tự
results = await asyncio.gather(
    fetch("A"), fetch("B"), fetch("C"),
    return_exceptions=True   # không fail toàn bộ khi 1 cái lỗi
)

# wait — lấy cái nào xong trước
tasks = [asyncio.create_task(fetch(u)) for u in urls]
done, pending = await asyncio.wait(
    tasks,
    return_when=asyncio.FIRST_COMPLETED   # dừng ngay khi có 1 xong
)
for task in pending:
    task.cancel()   # hủy các task còn lại
06. Generator là gì? Khi nào nên dùng?

Generator là hàm dùng yield thay vì return. Thay vì tính hết rồi trả về một lần, generator tạo từng phần tử theo yêu cầu — chỉ khi iterator gọi next(). Bộ nhớ gần như không đổi dù xử lý triệu dòng.

# ❌ Không dùng generator — 10M dòng vào RAM cùng lúc
def read_all(path):
    return open(path).readlines()   # ~1GB RAM với file lớn

# ✅ Generator — chỉ giữ 1 dòng tại 1 thời điểm
def read_lines(path):
    with open(path) as f:
        for line in f:
            yield line.strip()

for line in read_lines("huge.csv"):
    process(line)   # RAM không tăng

# Generator expression — còn ngắn hơn
squares = (x**2 for x in range(10_000_000))   # lazy, không tính ngay
total = sum(squares)   # tính từng cái, không cần list

Dùng generator khi: đọc file lớn, stream dữ liệu từ API, pipeline xử lý nhiều bước, hoặc chuỗi vô hạn.

07. __slots__ dùng để làm gì?

Mặc định mỗi object Python có __dict__ — từ điển nội bộ lưu thuộc tính, tốn khoảng 200–400 byte overhead. Khi khai báo __slots__, Python không tạo __dict__ nữa, lưu trực tiếp vào cấu trúc cố định — tiết kiệm 40–60% memory mỗi object.

import sys

class Transaction:
    def __init__(self, id, amount):
        self.id = id
        self.amount = amount

class TransactionSlots:
    __slots__ = ['id', 'amount']
    def __init__(self, id, amount):
        self.id = id
        self.amount = amount

t1 = Transaction(1, 100)
t2 = TransactionSlots(1, 100)

print(sys.getsizeof(t1) + sys.getsizeof(t1.__dict__))  # ~360 bytes
print(sys.getsizeof(t2))                                 # ~56 bytes

# 1 triệu object: ~304MB vs ~56MB
# Nhược điểm: không thể thêm thuộc tính ngoài __slots__

Dùng khi tạo hàng triệu object cùng kiểu: cache entry, event record, transaction row.

08. Decorator là gì? Cách hoạt động?

Decorator là hàm nhận một hàm làm input, bọc thêm logic, trả về hàm mới. Ký hiệu @decorator là cú pháp sugar: func = decorator(func).

Decorator như lớp áo ngoài — thêm tính năng mà không sửa bên trong.
import functools, time

def timer(func):
    @functools.wraps(func)
    def wrapper(*args, **kwargs):
        start = time.perf_counter()
        result = func(*args, **kwargs)
        elapsed = time.perf_counter() - start
        print(f"{func.__name__} took {elapsed:.3f}s")
        return result
    return wrapper

def retry(times=3):
    def decorator(func):
        @functools.wraps(func)
        async def wrapper(*args, **kwargs):
            for attempt in range(times):
                try:
                    return await func(*args, **kwargs)
                except Exception as e:
                    if attempt == times - 1:
                        raise
        return wrapper
    return decorator

@timer
def process_payment(amount): ...

@retry(times=3)
async def call_bank_api(payload): ...

Trong thực tế: logging, timing, retry, auth check, cache, rate limit — tất cả đều dùng decorator để tách logic cross-cutting ra khỏi business logic.

09. functools.wraps dùng để làm gì?

Khi tạo decorator, hàm wrapper che mất tên, docstring, signature của hàm gốc. functools.wraps sao chép metadata đó từ hàm gốc sang wrapper — quan trọng cho logging, debugging, và FastAPI (dùng tên hàm để generate route docs).

import functools

# ❌ Không dùng wraps
def log(func):
    def wrapper(*args, **kwargs):
        return func(*args, **kwargs)
    return wrapper

@log
def transfer(): "Transfer money"

print(transfer.__name__)   # "wrapper"   ← sai
print(transfer.__doc__)    # None         ← mất docstring

# ✅ Dùng wraps
def log(func):
    @functools.wraps(func)
    def wrapper(*args, **kwargs):
        return func(*args, **kwargs)
    return wrapper

@log
def transfer(): "Transfer money"

print(transfer.__name__)   # "transfer"  ← đúng
print(transfer.__doc__)    # "Transfer money"
10. Context manager là gì? Tại sao cần thiết?

Context manager đảm bảo tài nguyên được dọn dẹp đúng cách dù có exception hay không — thông qua cú pháp with ... as. Python gọi __enter__ khi vào block và __exit__ khi thoát ra (kể cả khi lỗi).

# ❌ Không có context manager — file có thể không được đóng khi lỗi
f = open("data.txt")
data = f.read()   # nếu lỗi ở đây → f.close() không bao giờ chạy
f.close()

# ✅ Context manager — luôn đóng dù có lỗi
with open("data.txt") as f:
    data = f.read()

# Tự tạo context manager cho DB transaction
class DBTransaction:
    def __enter__(self):
        self.conn = get_connection()
        self.conn.begin()
        return self.conn

    def __exit__(self, exc_type, exc_val, exc_tb):
        if exc_type:
            self.conn.rollback()   # lỗi → rollback
        else:
            self.conn.commit()     # OK → commit
        self.conn.close()          # luôn đóng

with DBTransaction() as conn:
    conn.execute("UPDATE accounts SET balance = balance - 100 WHERE id=1")
    conn.execute("UPDATE accounts SET balance = balance + 100 WHERE id=2")
11. Phân biệt deepcopy và shallow copy?

Shallow copy: tạo object mới ở tầng ngoài cùng, nhưng các object lồng bên trong vẫn là tham chiếu chung với bản gốc. Thay đổi bên trong → ảnh hưởng bản gốc.

Deep copy: đệ quy sao chép tất cả mọi thứ bên trong. Hoàn toàn độc lập với bản gốc.

import copy

original = {"user": {"name": "Iron", "balance": 1000}}

# Shallow copy — bug thực tế
shallow = copy.copy(original)
shallow["user"]["balance"] = 0
print(original["user"]["balance"])   # ❌ 0 — bản gốc bị sửa!

# Deep copy — an toàn
deep = copy.deepcopy(original)
deep["user"]["balance"] = 0
print(original["user"]["balance"])   # ✅ 1000 — bản gốc nguyên vẹn

# List cũng vậy
a = [[1, 2], [3, 4]]
b = a.copy()          # shallow
b[0].append(99)
print(a[0])           # [1, 2, 99] ← a bị ảnh hưởng!

Dùng deepcopy khi cần clone cấu hình để thử nghiệm, hoặc bất kỳ khi nào dữ liệu có cấu trúc lồng nhau và cần hoàn toàn độc lập.

12. List comprehension và hàm map() — khi nào dùng cái nào?

List comprehension: tạo list ngay, dễ đọc, Pythonic — ưu tiên dùng hầu hết trường hợp.

map(): trả về iterator lazy — không tính ngay, chỉ tính khi cần. Tiết kiệm RAM khi chuỗi rất dài và chỉ duyệt qua một lần.

data = range(1_000_000)

# List comprehension — rõ ràng, kết quả sẵn trong RAM
evens_squared = [x**2 for x in data if x % 2 == 0]

# map/filter — lazy, không tạo list trung gian
result = map(lambda x: x**2, filter(lambda x: x % 2 == 0, data))
total = sum(result)   # chỉ khi sum chạy mới tính từng phần tử

# Generator expression — lazy như map nhưng dễ đọc hơn
total = sum(x**2 for x in data if x % 2 == 0)   # ✅ best of both

Khuyến nghị: dùng list comprehension khi cần list thật, dùng generator expression khi chỉ cần duyệt qua một lần. map() ít dùng hơn trong Python hiện đại.

13. *args và **kwargs là gì?

*args: gom tất cả positional arguments thừa vào một tuple.
**kwargs: gom tất cả keyword arguments thừa vào một dict.
Dùng nhiều nhất trong decorator để truyền nguyên xi mọi đối số xuống hàm gốc.

import functools

def log(func):
    @functools.wraps(func)
    def wrapper(*args, **kwargs):   # nhận MỌI đối số, bất kể bao nhiêu
        print(f"Calling {func.__name__} args={args} kwargs={kwargs}")
        return func(*args, **kwargs)   # truyền xuống nguyên xi
    return wrapper

@log
def transfer(from_acc, to_acc, amount, currency="VND"):
    pass

transfer("A001", "B002", 500)
# "Calling transfer args=('A001','B002',500) kwargs={}"
transfer("A001", "B002", 500, currency="USD")
# "Calling transfer args=('A001','B002',500) kwargs={'currency':'USD'}"

# *args trong hàm thông thường
def sum_all(*args):
    return sum(args)

sum_all(1, 2, 3, 4, 5)   # 15
14. Python quản lý bộ nhớ như thế nào?

Ba cơ chế chính:

1. Reference counting: mỗi object có bộ đếm số nơi đang trỏ đến. Khi về 0, giải phóng ngay.

2. Cyclic garbage collector: xử lý circular reference — trường hợp 2 object trỏ nhau, ref count không bao giờ về 0 dù không ai dùng.

3. Memory pool: Python không xin/trả OS từng byte nhỏ (tốn kém). Thay vào đó giữ pool nội bộ, tái sử dụng ô nhớ đã giải phóng.

import sys

x = [1, 2, 3]
print(sys.getrefcount(x))   # 2 (x + tham số getrefcount)

y = x                        # ref count → 3
del x                        # ref count → 2
del y                        # ref count → 0 → giải phóng ngay

# Circular reference — cần GC
a, b = {}, {}
a['b'] = b
b['a'] = a     # a và b trỏ nhau, ref count không về 0
del a, b       # GC định kỳ sẽ phát hiện và xóa

import gc
gc.collect()   # ép chạy GC ngay nếu cần
15. is và == khác nhau thế nào?

==: so sánh giá trị — gọi __eq__.
is: so sánh danh tính — kiểm tra cùng địa chỉ bộ nhớ (id(a) == id(b)).

# Bug thực tế — nhầm is với ==
a = 1000
b = 1000
print(a == b)   # ✅ True  (giá trị bằng nhau)
print(a is b)   # ❌ False (hai object khác nhau trong RAM)

# Python cache int từ -5 đến 256 → is có thể True nhưng không nên dựa vào
x = 256; y = 256
print(x is y)   # True (cached)
x = 257; y = 257
print(x is y)   # False (không cached)

# String interning cũng gây nhầm lẫn
s1 = "hello"
s2 = "hello"
print(s1 is s2)   # True (interned) — nhưng đừng dựa vào điều này

# ✅ Quy tắc thực tế: is CHỈ dùng với None, True, False
if result is None: ...
if flag is True: ...   # hoặc đơn giản là: if flag: ...
16. Exception chaining (raise ... from ...) là gì?

Khi bắt lỗi kỹ thuật và muốn throw lỗi nghiệp vụ, dùng raise NewError from original. Python giữ cả hai trong traceback — người debug thấy đủ context, người dùng API thấy lỗi có nghĩa.

class PaymentError(Exception):
    pass

def get_account(account_id: str):
    try:
        return db.query("SELECT * FROM accounts WHERE id = %s", account_id)
    except DatabaseConnectionError as e:
        # Wrap lỗi kỹ thuật thành lỗi nghiệp vụ
        raise PaymentError(f"Cannot retrieve account {account_id}") from e

# Traceback sẽ hiện:
# DatabaseConnectionError: connection refused
# The above exception was the direct cause of:
# PaymentError: Cannot retrieve account ACC001

# raise ... from None: ẩn lỗi gốc hoàn toàn
raise PaymentError("Account not found") from None
17. Phân biệt classmethod, staticmethod và instance method?

Instance method: nhận self, truy cập thuộc tính object cụ thể.
classmethod: nhận cls, dùng làm factory — tạo object từ nhiều nguồn khác nhau.
staticmethod: không nhận self/cls, là hàm tiện ích đặt trong class cho gọn.

import json
from decimal import Decimal

class Payment:
    def __init__(self, amount: Decimal, currency: str):
        self.amount = amount
        self.currency = currency

    # Instance method — làm việc trên object cụ thể
    def display(self) -> str:
        return f"{self.amount} {self.currency}"

    # classmethod — factory: tạo object từ nhiều dạng input
    @classmethod
    def from_dict(cls, data: dict) -> "Payment":
        return cls(Decimal(str(data["amount"])), data["currency"])

    @classmethod
    def from_json(cls, json_str: str) -> "Payment":
        return cls.from_dict(json.loads(json_str))

    # staticmethod — validation, không cần self/cls
    @staticmethod
    def is_valid_amount(amount: float) -> bool:
        return 0 < amount <= 1_000_000_000

p = Payment.from_dict({"amount": "99.50", "currency": "USD"})
p = Payment.from_json('{"amount": "99.50", "currency": "USD"}')
Payment.is_valid_amount(-100)   # False
18. __init__ và __new__ khác nhau thế nào?

__new__: cấp phát bộ nhớ và tạo object trống — chạy trước.
__init__: điền thuộc tính vào object vừa tạo — chạy sau.
99% trường hợp chỉ cần __init__. __new__ dùng khi cần kiểm soát việc có tạo object mới hay không — ví dụ Singleton.

class Singleton:
    _instance = None

    def __new__(cls):
        # __new__ quyết định: tạo mới hay trả object cũ
        if cls._instance is None:
            cls._instance = super().__new__(cls)
        return cls._instance

    def __init__(self):
        if not hasattr(self, 'initialized'):
            self.config = {}
            self.initialized = True

s1 = Singleton()
s2 = Singleton()
print(s1 is s2)   # True — cùng 1 object

# __new__ cũng dùng khi kế thừa immutable type
class PositiveInt(int):
    def __new__(cls, value):
        if value <= 0:
            raise ValueError("Must be positive")
        return super().__new__(cls, value)   # int là immutable, phải dùng __new__
19. Pydantic dùng để làm gì? Tại sao quan trọng trong API tài chính?

Pydantic tự động validate và convert dữ liệu dựa trên type hints. Trong fintech, dữ liệu sai (số tiền âm, thiếu trường, sai định dạng) phải bị chặn ngay tại cổng vào — không để lọt vào business logic.

from pydantic import BaseModel, field_validator
from decimal import Decimal

class TransferRequest(BaseModel):
    from_account: str
    to_account: str
    amount: Decimal       # tự convert string "99.5" → Decimal
    currency: str = "VND"

    @field_validator("amount")
    @classmethod
    def must_be_positive(cls, v):
        if v <= 0:
            raise ValueError("amount must be positive")
        return v

    @field_validator("from_account", "to_account")
    @classmethod
    def must_not_be_same(cls, v, info):
        # validator chạy sau khi field được set
        return v

# FastAPI tự động validate — không cần code thêm
@app.post("/transfer")
async def transfer(req: TransferRequest):
    # req.amount đã là Decimal, đã validated dương
    await process(req.from_account, req.to_account, req.amount)

# {"amount": -100} → 422 Unprocessable Entity (tự động)
# {"amount": "abc"} → 422 (Decimal không parse được)
# {"amount": "50.5", ...} → ✅ req.amount = Decimal("50.5")

Pydantic cũng tự sinh OpenAPI docs — không cần viết tài liệu thủ công.

20. Làm sao tối ưu performance cho Python service xử lý nhiều request?

Năm hướng thực tế, theo thứ tự ưu tiên:

# 1. Async I/O — không block event loop
@app.get("/rates")
async def get_rates():
    data = await redis.get("exchange_rates")   # async, không block
    return data
